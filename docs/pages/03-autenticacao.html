📋 Visão Geral

O sistema de autenticação implementa fluxos completos de registro e login usando JWT (JSON Web Tokens), com hash de senhas via bcrypt, persistência no PostgreSQL via Prisma, e integração com Socket.IO para autenticação em tempo real.

🔒 Bcrypt

Hash de senhas com salt rounds = 10

🛡️ Middlewares de Autenticação

📡 HTTP (REST API)

export const auth = fastifyPlugin(async (app) => {
  app.addHook('preHandler', async (request) => {
    request.getCurrentUserId = async () => {
      try {
        const { sub } = await request.jwtVerify<{ sub: string }>();
        return sub;
      } catch {
        throw new UnauthorizedError('Invalid or expired token');
      }
    };

    request.getCurrentUserRole = async () => {
      try {
        const decoded = await request.jwtVerify<{ sub: string; role?: string }>();
        return decoded.role || 'USER';
      } catch {
        throw new UnauthorizedError('Invalid or expired token');
      }
    };
  });
});

🔌 Socket.IO Handshake

export async function authenticate(
  socket: Socket,
  next: (err?: Error) => void
): Promise {
  try {
    const token = socket.handshake.auth.token;

    if (!token) {
      return next(new Error('Authentication token required'));
    }

    const decoded = jwt.verify(token, env.JWT_SECRET) as JWTPayload;

    // Attach user info to socket
    (socket as any).user_id = decoded.sub;

    next();
  } catch (error) {
    next(new Error('Invalid or expired token'));
  }
}

💻 Frontend - Zustand + localStorage

Auth Store (Zustand)

export const useAuthStore = create()(
  persist(
    (set) => ({
      user: null,
      token: null,
      isAuthenticated: false,
      isLoading: false,
      isHydrated: false,

      setAuth: (user, token) => {
        localStorage.setItem('token', token)  // ⚠️ XSS Risk!
        set({ user, token, isAuthenticated: true, isLoading: false })
      },

      logout: () => {
        localStorage.removeItem('token')
        set({ user: null, token: null, isAuthenticated: false })
      },

      setHydrated: () => set({ isHydrated: true }),
    }),
    {
      name: 'auth',
      partialize: (state) => ({ token: state.token }),
      onRehydrateStorage: () => (state) => {
        state?.setHydrated()
      },
    }
  )
)

⚠️ Observação: O token JWT é armazenado no localStorage, que é vulnerável a ataques XSS. Em produção, considere usar httpOnly cookies.

⚠️ Riscos de Segurança Identificados

1. JWT no localStorage (XSS)

O token JWT é armazenado no localStorage, que é acessível via JavaScript. Um ataque XSS pode roubar o token e dar acesso total à conta.

Solução recomendada: Usar httpOnly cookies com SameSite=Strict/Lax

2. Sem Refresh Token

O JWT expira em 1 hora, mas não há refresh token. O usuário precisa fazer login novamente após expirar. Isso prejudica a UX e força reautenticação frequente.

Solução: Implementar refresh tokens com maior expiração (7 dias)

3. Sem Blacklist/Revogação

Não há forma de invalidar um JWT antes da expiração. Se o token for comprometido, válido até expirar.

Solução: Manter blacklist de tokens revogados no Redis (IAT check)

4. Sem Rate Limiting por IP

Os endpoints /auth/register e /auth/login não têm rate limit. Isso permite ataques de força bruta e spam de criação de contas.

Solução: Adicionar rate limit por IP (ex: 5 tentativas/minuto)

5. Senhas em Texto Plano na Query

Na rota de login, o schema Zod permite qualquer string, mas não há validação de complexidade (maiúsculas, números, símbolos). Apenas min 6 caracteres.

Solução: Aumentar mínimo e adicionar regras: upper, number, specialChar

🔍 Comparação: Sessão Tradicional vs JWT

Característica Tradicional (Session) JWT (Implementado)
Armazenamento Server-side (Redis/DB) Client-side (token)
Estado Stateful Stateless
Scalability Requer compartilhamento de sessão Fácil escalonamento
Revogação Imediata (delete session) Apenas após expiração
Tamanho payload ID da sessão pequeno Maior (dados incluídos)
CSRF Vulnerável Imune (não usa cookies)

✅ Recomendações de Melhoria

1

Substituir localStorage por httpOnly cookies

Armazene o JWT em cookies httpOnly para proteção contra XSS. Use SameSite=Strict para evitar CSRF.

2

Implementar refresh tokens

Short-lived access tokens (15min) + long-lived refresh tokens (7 dias). Refresh endpoint para renovação.

3

Adicionar rate limiting

Limitadores por IP e por email nos endpoints de auth. Use Redis para contadores distribuídos.

4

Blacklist de tokens revogados

Armazene JTI (JWT ID) dos tokens invalidados no Redis com TTL igual à expiração. Verificar em middleware.

5

Política de senhas mais forte

Min 8+ chars, uppercase, numbers, symbols. Validar com regex ou biblioteca como validator.js.

6

2FA (Two-Factor Authentication)

Adicionar segunda camada com TOTP (Google Authenticator) ou codes por email/SMS.

📐 Schema do Banco - Model User

model User {
  id               String    @id @default(cuid())
  name             String?
  email            String    @unique
  password         String
  role             Role       @default(USER)
  isActive         Boolean    @default(true)
  lastLoginAt      DateTime?
  createdAt        DateTime   @default(now())
  updatedAt        DateTime   @updatedAt

  // Relations
  messages         Message[]
  rooms           RoomMember[]
  createdRooms    Room[]     @relation("RoomCreator")
}

enum Role {
  USER
  ADMIN
  MODERATOR
}
