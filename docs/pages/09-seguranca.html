A01: Broken Access Control

POST /users sem verificação de role explícita. Usuários podem criar outros usuários sem restrição de permissão. 

// backend-boilerplate/src/http/routes/user/create-user.ts
app
  .register(auth) // ✅ Autenticado, mas sem verificação de role
  .post('/users', async (request, reply) => {
    const { role } = request.body; // ⚠️ Role do usuário pode ser qualquer coisa
    // Qualquer usuário autenticado pode criar outros usuários
    const user = await prisma.user.create({
      data: { role, ... } // ⚠️ Sem verificação se é ADMIN
    });
  });

Recommendation: Middleware de autorização (ex: requireRole(['ADMIN'])) ou verificação no handler: if (user.role !== 'ADMIN') throw new ForbiddenError().

A03: Injection

Prisma ORM usa prepared statements automaticamente. Sem concatenação de SQL. 

// Prisma - safe por padrão
const user = await prisma.user.findUnique({
  where: { email } // ✅ Parametrizado
});

const user = await prisma.user.create({
  data: { name, email, password: hash } // ✅ Safe
});

Se usar raw queries, sempre utilize $params do Prisma para parametrização.

OK

A04: Insecure Design

JWT no localStorage → vulnerável a XSS. Se site tiver XSS, attacker lê token. 

// ❌ Vulnerável a XSS
localStorage.setItem('token', response.token);

// ✅ Melhor: httpOnly cookie (set via server)
// ou Redux/Zustand store (menos acessível que localStorage)
// mas ainda acessível via window.__REDUX_STATE__ se injetado

Mitigação: usar cookies httpOnly, secure, sameSite=strict. Implementar Content Security Policy forte.

A05: Security Misconfiguration

CORS habilitado em produção sem CORS_ORIGINS definido (default: desativado). Swagger sem proteção se credenciais não configuradas. 

// server.ts - CORS
const allowedOrigins = env.CORS_ORIGINS
  ? env.CORS_ORIGINS.split(',').map(o => o.trim())
  : env.NODE_ENV === 'production'
    ? []  // ✅ Desativado em prod por padrão
    : ['http://localhost:5173', 'http://localhost:4000'];

app.register(fastifyCors, {
  origin: allowedOrigins.length > 0 ? allowedOrigins : true,
  credentials: true,
});
// server.ts - Swagger sem proteção
const swaggerUser = env.SWAGGER_USER;
const swaggerPass = env.SWAGGER_PASSWORD;

if (swaggerUser && swaggerPass) {
  app.register(fastifyBasicAuth, { ... });
} else {
  console.warn('⚠️ Swagger is not protected - set SWAGGER_USER...');
}

CORS ok se CORS_ORIGINS definido em produção. Swagger deve ter basic-auth em prod.

🛡️ Configuração Helmet (server.ts)

Helmet adiciona headers de segurança HTTP automaticamente. 

// server.ts - Linhas 111-128
app.register(fastifyHelmet, {
  // Content Security Policy
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],              // Apenas próprio domínio
      styleSrc: ["'self'", "'unsafe-inline'"], // Inline styles necessários
      scriptSrc: ["'self'"],               // Apenas scripts próprios
      imgSrc: ["'self'", "data:", "https:"], // Imagens e data URIs
    },
  },

  // HSTS - Força HTTPS
  hsts: {
    maxAge: 31536000,          // 1 ano
    includeSubDomains: true,  // Inclui subdomínios
    preload: true,            // Pode ser preloaded no Chrome
  },

  // Clickjacking protection
  frameguard: { action: 'deny' },  // Bloqueia iframes

  // DNS prefetch control
  dnsPrefetchControl: { allow: false }, // Desabilita dns-prefetch
});

Nota: CSP pode ser ajustada conforme frontend precisa (ex:允许 CDN Tailwind, Google Fonts). Para produção rigorosa, remover unsafe-inline e usar nonces/hashes.

🔀 CORS Configuration

CORS restritivo baseado em variável de ambiente. Em produção, desativado por padrão se CORS_ORIGINS não definido. 

// server.ts - Linhas 141-152
const allowedOrigins = env.CORS_ORIGINS
  ? env.CORS_ORIGINS.split(',').map(o => o.trim())
  : env.NODE_ENV === 'production'
    ? []  // ❌ Production: CORS desativado sem whitelist explícita
    : ['http://localhost:5173', 'http://localhost:4000']; // Dev: localhost allowed

app.register(fastifyCors, {
  origin: allowedOrigins.length > 0 ? allowedOrigins : true,
  // Se array vazia em prod → origin: false (CORS bloqueado)
  credentials: true,
  methods: ['GET', 'POST', 'PUT', 'DELETE', 'PATCH', 'OPTIONS'],
});

✅ Padrão Seguro

Prod sem CORS_ORIGINS → CORS bloqueado. Dev localhost liberado.

⚠️ Config Necessária

Em prod, setar CORS_ORIGINS=https://meuapp.com,https://app.meuapp.com no .env.

🔐 JWT Configuration

JWT configurado com secret no .env e expiração de 1 hora. Sem refresh token (⚠️ crítico). 

// server.ts - Linhas 216-219
app.register(fastifyJwt, {
  secret: env.JWT_SECRET,  // ⚠️ Deve ser forte e em .env
});

// authenticate.ts - Linhas 66-69
const token = await reply.jwtSign(
  { sub: user.id },
  { expiresIn: '1h' }  // ✅ Expiração curta (apenas 1h)

❌ Missing: Refresh Token

Sem refresh token, usuário precisa logar novamente após 1h. Implementar refresh tokens com blacklist no Redis para revogação. Exemplo: refresh_token com expiração de 7d, armazenado em httpOnly cookie, válido apenas para troca por novo access_token.

⚠️ Brute-force

Rate limit atual é por IP. Adicionar rate limit por userId para evitar bloqueio de contas.

⚠️ 2FA

Sem Two-Factor Authentication. Implementar TOTP com speakeasy para contas ADMIN ou opcional para todos.

📁 File Upload Security

fastifyMultipart configurado com limits. Falta validação de tipo MIME e scan de malware. 

// server.ts - Linhas 155-161
app.register(fastifyMultipart, {
  limits: {
    fileSize: env.MAX_FILE_SIZE, // ✅ Limite configurável
    files: 5,     // ✅ Máximo 5 arquivos por request
    fields: 50,   // ✅ Máximo 50 campos
  },
});

❌ Missing: MIME Validation & Malware Scan

Qualquer tipo de arquivo pode ser enviado se dentro do size limit. Ataque: upload de .php ou .exe, se execução possível.

Solução: Validar MIME com file-type (não confiar em Content-Type header) e escanear com ClamAV se需 alta segurança. 

// Exemplo de validação MIME
import { fileTypeFromStream } from 'file-type';

const fileType = await fileTypeFromStream(req.payload);
if (!['image/jpeg', 'image/png', 'application/pdf'].includes(fileType.mime)) {
  throw new BadRequestError('Tipo de arquivo não permitido');
}

💉 SQL Injection Protection

Prisma usa prepared statements automaticamente. Sem risco de SQL Injection com query builder. 

// ✅ SAFE - Prepared statements automáticos
const user = await prisma.user.findUnique({
  where: { email }  // Parâmetro parametrizado
});

// ✅ SAFE - Prisma valida tipos
const user = await prisma.user.create({
  data: {
    email,  // 🛡️ Escapado
    name,
  }
});

// ⚠️ CUIDADO se usar raw queries:
await prisma.$executeRaw`SELECT * FROM users WHERE email = ${email}`;
// ✅ OK usando template literal do Prisma (parametriza)

// ❌ NUNCA fazer:
await prisma.$executeRaw(`SELECT * FROM users WHERE email = '${email}'`);
// SQL Injection possível

✅ Prisma é seguro por padrão. Apenas evitar concatenação de strings em $executeRaw ou $queryRaw.

🔑 Redis Security

Redis pode ter senha configurada via .env. TLS opcional mas recomendado em produção. 

// .env
REDIS_URL=localhost
REDIS_PORT=6379
REDIS_PASSWORD=  # ⚠️ Senha opcional mas recomendada

// server.ts - Linhas 222-228
if (redisAvailable) {
  app.register(fastifyRedis, {
    host: env.REDIS_URL,
    password: env.REDIS_PASSWORD || undefined,  // ✅ Senha usada se definida
    port: env.REDIS_PORT,
    family: 4,
  });
}

Melhoria: Em produção, usar Redis com TLS (rediss:// ou tls: {} no ioredis) e always senha.

⏱️ Rate Limiting

Rate limit global 100 req/min via Redis. Se Redis indisponível, limitação cai (degraded mode). 

// server.ts - Linhas 130-139
app.register(fastifyRateLimit, {
  max: 100,              // 100 requests
  timeWindow: '1 minute', // por minuto
  redis: redisAvailable ? app.redis : undefined, // ⚠️ Sem Redis → sem rate limit
  keyGenerator: (request) => {
    return request.ip;  // ✅ Por IP
    // TODO: por userId se autenticado
  },
});

Limitação: Se Redis cair, rate limit não funciona. Considerar fallback em memória (concurrent requests) ou reject imediato se Redis falhou? Atualmente degrada silenciosamente.

📊 Security Ranking & Priorities

🔴 Critical / High

  • A01 Broken Access Control: POST /users sem verificação de role → implementar middleware requireRole('ADMIN')
  • A07 Auth Failure: Sem refresh token, 2FA ou brute-force → criar sistema de refresh tokens com blacklist Redis
  • A04 Insecure Design: JWT localStorage → migrar para httpOnly cookies
  • MIME Validation: Uploads sem validação → implementar file-type check

🟡 Medium

  • A05 Misconfig: Swagger sem proteção em prod → ativar basic-auth
  • A09 Logging: Sem requestId tracing → integrar logging estruturado
  • Rate Limit fallback: Sem Redis → rate limit para de funcionar → implementar memória fallback
  • Redis TLS: Sem TLS em produção → configurar tls: {}

🟢 Low / OK

  • A02 Crypto: Bcrypt e HTTPS ok
  • A03 Injection: Prisma safe
  • A08 Data Integrity: Zod + Prisma validation
  • A10 SSRF: Sem user-controlled URLs
  • .env protection: gitignore configurado
  • Helmet: HSTS, CSP, frameguard ativos

🚀 Recomendações de Melhoria

  2. Implementar refresh tokens com blacklist Redis
  3. Ativar basic-auth no Swagger em produção

  1. Implementar HTTP-only cookies para JWT (eliminar localStorage)
  2. Adicionar validação MIME em uploads
  3. Criar rate limit por userId (além de IP)
  4. Implementar 2FA com TOTP

Médio Prazo (P2)

  1. Adicionar logging estruturado com requestId

  4. Refinar CSP (remover unsafe-inline, usar nonces)

Contínuo
