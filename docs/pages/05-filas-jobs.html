🏃 Workers

worker-manager.ts — Gerenciador Principal

Inicia todos os workers da aplicação via startAllWorkers(), chamado em server.ts após o Boot do Redis. Cada worker é indexado por nome de fila em um Map<string, WorkerInstance>.

Lifecycle hooks por worker:
  • completed — loga tempo de execução
  • failed — move para DLQ após esgotar tentativas
  • stalled — loga job travado
  • error — loga erros do processo worker

example-worker.ts — Worker de Referência

Template minimalista para criação de novos workers. Processa 1 job por vez na fila example-queue com concorrência padrão 5. Demonstra o padrão de startWorker() / stopWorker(). 

// example-worker.ts — fluxo simplificado
async function processExampleJob(job: Job) {
  console.log(`Processing job ${job.id}`);
  // lógica do job aqui
  await doWork(job.data);
  return { success: true };
}

// Cria worker conectado ao Redis
worker = new Worker('example-queue', processExampleJob, {
  concurrency: 5,
  connection: connectionRedisConfigWorker,
});

// Eventos
worker.on('completed', (job) => { /* ... */ });
worker.on('failed',    (job, err) => { /* ... */ });

Note: example-worker.ts usa a fila example-queue que não faz parte do QUEUE_NAMES oficial — é apenas um template de desenvolvimento.

🔴 Conexão Redis (connection-redis-config.ts)

Configurações de Conexão

// Para Queues (Producer)
export const connectionRedisConfigQueue = {
  host: env.REDIS_URL,
  password: env.REDIS_PASSWORD,
  port: env.REDIS_PORT,
  family: 4, // força IPv4
};

// Para Workers (Consumer)
export const connectionRedisConfigWorker = {
  maxRetriesPerRequest: null, // obrigatório BullMQ
  host: env.REDIS_URL,
  password: env.REDIS_PASSWORD,
  port: env.REDIS_PORT,
  family: 4,
  connectTimeout: 10000, // 10s
};

maxRetriesPerRequest: null é obrigatório pelo BullMQ para workers — permite que o ioredis retenha comandos pendentes indefinidamente em vez de falhar imediatamente.

Modo Degradado (Redis Indisponível)

O redisInstance expõe isDegraded(). Quando Redis cai:

  • ✗ getQueue(name) retorna null — jobs não são enfileirados
  • ✗ startWorker(config) retorna null — workers não sobem
  • ⚠ Rate limiter Redis cai graciosamente para RateLimiterMemory
  • ⚠ Bull Board não registra adaptadores de fila — painel fica vazio
  • ✓ Servidor HTTP continua operando para requisições síncronas
Atenção: Jobs não enfileirados durante modo degradado são perdidos — não há persistência local nem replay automático ao Redis se recuperar.

📊 Bull Board Dashboard

Configuração em server.ts

import { createBullBoard }   from '@bull-board/api';
import { BullMQAdapter }     from '@bull-board/api/bullMQAdapter';
import { FastifyAdapter }    from '@bull-board/fastify';

const serverAdapter = new FastifyAdapter();

createBullBoard({
  queues: queueAdapters, // todas as filas ativas
  serverAdapter,
});

serverAdapter.setBasePath('/queues');
app.register(serverAdapter.registerPlugin(), {
  prefix: '/queues',
});

// Acessível em: http://host:PORT/queues

O Bull Board só é registrado se o Redis estiver disponível e ao menos uma fila estiver ativa (queueAdapters.length > 0).

Autenticação

O servidor usa @fastify/basic-auth (importado em server.ts). O padrão atual aplica Basic Auth apenas ao Swagger (SWAGGER_USER / SWAGGER_PASSWORD). A rota /queues não possui autenticação configurada por padrão — é acessível sem credenciais em desenvolvimento.

Em produção, proteja /queues adicionando um hook de autenticação ou colocando o painel atrás de uma VPN/proxy.

Erro lançado; aguarda backoff

⏰

Aguardando backoff expirar para nova tentativa
→
☠️
dead letter
Todas as tentativas esgotadas; movido para DLQ
Stalled: Se o worker crasha com job ativo, BullMQ detecta stall após lockDuration (30s). Job volta para waiting. Máximo maxStalledCount: 2 vezes.
Delayed: Jobs com delay definido entram neste estado ao ser adicionados e só passam para waiting após o timer expirar.
removeOnComplete: Cada fila define count de jobs concluídos a manter. Emails: 500, Notificações: 1000, Background: 50.

💻 Exemplos de Código

addJob() — Enfileirar um job

import { addJob, QUEUE_NAMES } from
  '@/services/jobs/queue/queue-manager';

// Enfileirar e-mail simples
const job = await addJob(QUEUE_NAMES.EMAIL, {
  name: 'send-email',
  data: {
    to: 'user@example.com',
    subject: 'Bem-vindo!',
    template: 'welcome',
  },
  options: {
    delay: 0,       // imediato
    priority: 1,    // alta prioridade
    jobId: `email-${userId}`, // dedup
  },
});

// Enfileirar com delay (ex: lembrete 24h)
await addJob(QUEUE_NAMES.NOTIFICATION, {
  name: 'push-notification',
  data: { userId, title: 'Lembrete' },
  options: {
    delay: 24 * 60 * 60 * 1000, // 24h em ms
  },
});

// Parar worker (graceful: termina job ativo antes de fechar)

⚠️ Riscos e Problemas Conhecidos

💀 Stuck Jobs (Worker Crash)

Se um worker crasha enquanto processa um job, o job permanece no estado active até o lock expirar (30s). Após expirar, BullMQ marca como stalled e recoloca em waiting. Se ocorrer maxStalledCount: 2 vezes, o job é movido para failed.

Mitigação: Use lockRenewTime: 15000 para renovar lock a cada 15s em jobs longos. Para jobs >30s, aumente lockDuration proporcionalmente.

📈 Memory Growth

Jobs concluídos e falhos acumulam no Redis se removeOnComplete/removeOnFail não estiver configurado. O DEAD_LETTER usa removeOnComplete: false, removeOnFail: false — cresce indefinidamente.

Mitigação: Implementar job de limpeza periódica da DLQ via BACKGROUND queue. Monitorar MEMORY redis INFO em produção.

🔴 Redis Outage

Em modo degradado, todos os jobs são silenciosamente descartados — addJob() retorna null sem exceção. O código chamador precisa verificar o retorno para detectar falha.

Mitigação: Adicionar fallback de persistência local (SQLite/banco relacional) para jobs críticos durante outage. Verificar retorno de addJob() e logar/alertar quando null.

🔐 Bull Board sem Auth em Produção

O painel /queues está acessível publicamente por padrão. Expõe dados de jobs (incluindo payloads com e-mails, tokens, dados de usuários) e permite remover/reprocessar jobs manualmente.

Mitigação: Adicionar preHandler com Basic Auth (já disponível via @fastify/basic-auth) ou restringir à rede interna via reverse proxy.
