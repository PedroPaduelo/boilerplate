Em fluxos de exclusão/destruição com AlertDialog (Radix) + useMutation, FECHAR o diálogo sempre via onSettled — onSuccess deixa o overlay travado em falhas (rede/500/404). Use o hook compartilhado useConfirmDelete<T>. Anti-padrão ainda vivo em delete-connection-dialog e delete-user-dialog (issue aberta).

conteúdo completo 
**REGRA do projeto (frontend-boilerplate):** SEMPRE que uma `useMutation` do React Query for chamada dentro de um `AlertDialog` (Radix) e o resultado precisar FECHAR o diálogo, use `onSettled` — NUNCA `onSuccess`. O `onSuccess` só fecha em caso de sucesso, e qualquer falha (rede, 500, 404) deixa o `open` travado em `true`, bloqueando cliques no body (overlay Radix).

**Anti-padrão PROIBIDO (causa bug "UI trava após ação"):**
```tsx
mutation.mutate(id, { onSuccess: () => onOpenChange(false) });
// ou
mutation.mutate(id, { onSuccess: () => setDeleting(null) });
```

**Padrão correto (via `useConfirmDelete<T>` em `src/shared/hooks/use-confirm-delete.ts`):**
```tsx
const { dialogProps, openDelete } = useConfirmDelete<MyItem>({
  mutation: useDeleteMyItem(),
  getId: (item) => item.id,
  getTitle: (item) => item.title,
});

<ConfirmDeleteDialog title="Excluir?" {...dialogProps} />
```
O hook centraliza o state e fecha SEMPRE via `onSettled` da mutação.

**Onde o anti-padrão AINDA existe (issue cmqpr62ac00mvpi0ifi4504n6):**
- `src/features/connections/components/delete-connection-dialog.tsx` (linhas 24-28)
- `src/features/users/components/delete-user-dialog.tsx` (linhas 24-28)

Ambos montam `AlertDialog` diretamente (não usam `ConfirmDeleteDialog` compartilhado) e replicam o anti-padrão. Quando criar/modificar fluxos de exclusão, SEMPRE preferir o hook compartilhado. Follow-up: unificar esses 2 com `ConfirmDeleteDialog` + `useConfirmDelete`.

**Aplicável a:** qualquer nova feature que adicionar exclusão/destruição (departments, shares, MCP tools destrutivas no FE, etc.).
